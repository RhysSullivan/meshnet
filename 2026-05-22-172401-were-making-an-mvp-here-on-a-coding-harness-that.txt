  ---
  Meshnet — POC One-Pager

  Infinite VM-backed dev servers, multiplexed through one local origin

  The idea

  A coding harness where every chat owns a machine. Each machine runs the user's
   dev server on some port. The harness UI lives at localhost:3000 and embeds a
  per-chat in-app browser (à la Cursor). A reverse proxy multiplexes every
  browser window to the right machine's dev server — one origin, N backends,
  routed by hostname.

  ┌─ localhost:3000 (harness UI) ──────────────────────────┐
  │  Chat A  ◄─iframe─►  a.localhost ─┐                     │
  │  Chat B  ◄─iframe─►  b.localhost ─┤  portless proxy     │
  │  Chat C  ◄─iframe─►  c.localhost ─┘  (host-based route) │
  └──────────────────────────────────────┬─────────────────┘
                                          │ forwards each host to
                            ┌─────────────┴──────────────┐
                            ▼             ▼               ▼
                      Vercel Sandbox  Sandbox        Sandbox
                      (microVM,       (microVM)      (microVM)
                       dev on :3000)
                       sandbox.domain(3000) → public URL

  The two building blocks (researched)

  1. Your portless fork — the multiplex layer

  RhysSullivan/portless (fork of vercel-labs/portless). It's a Node HTTP/2 + TLS
   reverse proxy that maps stable <name>.localhost URLs to backend ports. The
  pieces that matter for us, already in the code:
  - Host-based routing (proxy.ts:findRoutes) — exact match plus wildcard 
  subdomain fallback (tenant.myapp.localhost → myapp.localhost). This is the
  multiplex primitive: one proxy, many named routes.
  - Routes carry rich metadata — { hostname, port, pid, cwd, folder, gitBranch, 
  command, id }. Maps cleanly onto "one route per chat/sandbox."
  - App switcher built in — SELECTOR_COOKIE (portless_app) + a /__portless__
  control prefix + HTML injection. Already most of an in-app "which machine am I
   looking at" selector.
  - HTTP/2, X-Forwarded-* headers, loop detection (x-portless-hops, max 5 hops).

  Key adaptation for the POC: upstream portless forwards to a local port. Our
  backends are remote sandboxes, so a route's target becomes the sandbox's
  public domain() URL instead of 127.0.0.1:<port>. That's the one real change to
   the proxy.

  2. Vercel Sandbox — the infinite-VM layer

  @vercel/sandbox — ephemeral Linux microVMs on demand, TypeScript SDK (and a
  CLI for quick prototyping).
  - Sandbox.create({ runtime: 'node24', ports: [3000], timeout }) — up to 15 
  ports, default timeout 5 min (extendable via extendTimeout).
  - sandbox.runCommand({ cmd, args, detached: true }) — start a long-running dev
   server without blocking.
  - sandbox.domain(3000) → a public preview URL for an exposed port. This is
  what the proxy forwards to.
  - Persistent by default — snapshots filesystem on stop, restores on resume.
  Sandbox.get(name) re-attaches (auto-resumes on next call). So "chat → machine"
   survives idle/restart.
  - Billed on CPU-ms + network per session; idle sandboxes stop and snapshot, so
   cost ≈ active usage.

  Proposed POC scope

  1. Harness shell at localhost:3000 (Bun.serve + React per CLAUDE.md): chat
  list + an iframe pane per chat.
  2. Sandbox manager: createChat() → Sandbox.create, clone/scaffold a template
  app, runCommand('dev', { detached }), capture sandbox.domain(port). Persist
  chatId → { sandboxName, domain }.
  3. Proxy bridge: register a portless route per chat (<chatId>.localhost →
  sandbox domain). Iframe src = that route.
  4. Lifecycle: idle chats let their sandbox stop+snapshot; reopening a chat
  calls Sandbox.get to resume. Surface status (pending|running|stopped) in the
  UI.

  Explicitly out of scope for the POC: auth, multi-user, custom domains/TLS
  niceties, monorepo discovery, real persistence beyond a JSON/SQLite map.

  Open questions before we build

  1. Proxy-to-remote vs. iframe-direct. Do we route remote sandboxes through
  portless (single localhost:3000 origin, cookie/header control, app switcher —
  but proxy must rewrite to remote URLs)? Or iframe the sandbox.domain() URLs
  directly and use portless only for local mode? The former matches your
  "everything on :3000" vision; the latter is faster to stand up.
  2. SDK vs CLI for sandbox control in the POC — SDK gives us lifecycle/status;
  CLI is faster to spike. Lean SDK?
  3. Template app each chat boots into (blank Next.js? Vite? something with an
  obvious port).

   (route-through-portless, SDK, blank Next.js) and start scaffolding?

❯ does it change if we're doing it as a desktop app like the codex desktop app

⏺ Yes — it changes everything, and in the good direction. The constraint I
  described is a browser sandbox limitation. In a desktop app (Electron, like
  Codex desktop / Cursor) you own the network layer, so it disappears.

  Why the constraint lifts

  In a plain browser, all iframes on localhost:3000 share one cookie jar / one
  origin, so you can't route them to different backends — the only discriminator
   that survives navigation is the origin itself.

  In Electron you route by which view made the request, not by URL. So every
  view can be localhost:3000 and still go to a different VM:

  - Each chat gets its own session partition: <webview 
  partition="persist:chat1">, partition="persist:chat2", … (or a WebContentsView
   per chat with its own session).
  - You intercept requests per partition via
  session.fromPartition('persist:chatN').protocol.handle('http', …) (or
  webRequest / a custom protocol).
  - The handler forwards that partition's localhost:3000 requests to that chat's
   Vercel Sandbox domain.

  Result:
  - All views are genuinely localhost:3000 — location.href, fetch('/api'),
  absolute asset paths, cookies, in-page navigation all just work, because each
  partition is a fully isolated origin instance.
  - Simultaneous + state-preserving: each view stays mounted and live; switching
   is instant, no reload, no counter reset.
  - No subdomains, no ports, no path prefixes, no injection hacks. The
  discriminator is the view/partition, invisible to the app.

  This is almost certainly how Codex desktop does its per-chat browser: a
  WebContentsView/webview per chat with an isolated session, intercepting at the
   app layer. The reverse-proxy multiplex we've been building in the browser is
  exactly the thing the desktop shell gives you for free.

  What it'd mean for the POC

  The Vercel Sandbox layer (sandboxes.ts) is unchanged — still Sandbox.create +
  domain(3000). What changes is the shell: drop the :3000 Bun reverse-proxy and
  the iframe URL gymnastics; replace with an Electron main process that maps
  partition → sandbox domain and intercepts requests. The harness UI (sidebar +
  browser chrome) becomes the renderer.
